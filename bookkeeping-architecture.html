Section 1
Core Concepts
Four primitives that compose the entire system.
🎯
Intent
The why / what / whom of a transaction. Determines which Strategy handles it. Currently a single field; a richer LLM-inferred model is in-progress.
routing_intent CharField — on Transaction purpose / medium / entity_type Planned confidence FloatField — LLM score

Strategy
A handler for a class of transactions needing specialised logic beyond COA assignment. Each strategy has its own config model and processing workflow.

Recipe
The account-level routing configuration for a CapitalAccount. Composed of RoutingRules + StrategyConfigs. Created at onboarding; refined over time. No single model — emergent from row composition.
RoutingRule conditions (JSONField) + FK → StrategyConfig StrategyConfig bridge → concrete config priority ASC first match wins
🔄
Workflow
The orchestration engine ("agent" in business). A node graph with a shared context blackboard. Nodes are tools or sub-workflows; transitions are condition-driven. Called "Agent" by the product team.
WorkflowNode foreground (human gate) | background (Celery) context Dict[str, Any] — serialised to Postgres workflow_class_path version-pinned at creation

Section 3
Recipe Architecture
Account-level configuration: which Intents map to which Strategies. No single model — emergent from RoutingRule + StrategyConfig composition.
RoutingRule
Condition matcher — priority-ordered, first match wins

Carries a conditions JSONField evaluated AND-style. Missing key = wildcard.

description_pattern counterparty_pattern direction: inflow | outflow amount_min / amount_max → strategy_config FK → target_module (fallback)
StrategyConfig
Bridge model — strategy_type derived from which FK is set

No strategy_type column in the DB. The type is inferred at runtime by inspecting which FK is non-null.

processor_config → ProcessorConfig (income_split) transfer_config → TransferConfig (transfer) check_config → CheckMatchConfig (check)
Lifecycle
Onboard → Apply → Refine

Phase 1: seed helpers create RoutingRule + StrategyConfig rows at client onboarding.
Phase 2: Every bank statement run evaluates rules priority-ASC; stamps routing_intent.
Phase 3: Admins add/edit RoutingRule rows as new transaction patterns emerge.

Section 5
Workflow Engine
Node-graph orchestrator. Called "Agent" in product discussions — each workflow is a named agent with composable tool-nodes.
Node Graph
Flat dictionary — not a DAG library. Nodes registered via add_node(), edges via add_transition(). First matching condition wins. None condition = unconditional edge. Self-loops handle pagination.
Context Blackboard
Single mutable dict shared by all nodes. Serialised to WorkflowExecution.context (JSONField) after every node inside an atomic() block. Enables crash-resume without re-running prior nodes.
Background vs Foreground
Background: execute_in_background=True → Celery task dispatched. Worker reconstructs workflow from workflow_class_path.
Foreground: synchronous / human gate — blocks until explicit input via API.
Versioning
AllowedWorkflow gates per-tenant access. At creation, workflow_class_path is stamped (immutable). Celery uses path-based lookup — immune to registry ordering and mid-run deployments.
# Version-stable Celery class resolution (workflows/tasks.py) workflow_class = ( registry.get_workflow_by_path(execution.workflow_class_path) if execution.workflow_class_path else None ) or registry.get_workflow(execution.workflow_name) # fallback

Section 7
Multi-tenancy & Operations
Every firm gets a dedicated Postgres schema. Celery queues are per-tenant. Workflow versions are pinned at creation.
Schema Isolation
django-tenants activates schema via HeaderTenantMiddleware reading X-Tenant-Schema. All ORM queries automatically hit the correct search path — no app-level tenant_id filter required.
Celery Queue Routing
Queue name: {app}-{env}-{schema}-{queue_suffix}
TenantAndTeamAwareTask injects schema_name + team_id into every payload. Worker restores both contexts before executing. Broker: Google Cloud Pub/Sub.
Section 8
Key Design Decisions
Grounded in the codebase — not aspirational.
Decision
Choice
Rationale
Rule matching order
Static (exact) → Dynamic (regex) → LLM (embedding)
Cheapest / most deterministic first. LLM only processes residual, minimising cost and latency.
Static match lock
"Rule Match Status" = "Single Match" — never overwritten
Prevents a weaker dynamic or LLM signal from degrading a deterministic exact match.
MongoDB append-only
Inserts only; current state = latest by created_at
Full version history retained; stale reads are safe; no migration drift on rule schema changes.
Postgres fallback cache
CapitalAccount.static_rules_cache / dynamic_rules_cache
MongoDB outage must not break the categorisation workflow. Self-healing write-back on first read.
workflow_class_path stamped at creation
Immutable on WorkflowExecution
Mid-flight executions are never affected by a new deployment registering a newer workflow version.
StrategyConfig bridge model
strategy_type derived from which FK is set — no enum column
Adding a new strategy = new concrete config model + new FK. No enum migration required.
LLM clustering: 4 rounds
0.85 → 0.90 → 0.95 → 0.98, force-accept on round 4
Coarse clusters handle obvious groups cheaply; tighter rounds resolve ambiguous descriptions; force-accept guarantees 100% coverage.
Section 9
Architecture Gaps & Roadmap
Honest gaps identified from the codebase — not from aspirational docs.
Intent Inference (InferIntentNode)
Branch not merged
Currently all transactions take the CATEGORISATION path. The richer TransactionIntent model (purpose / medium / entity_type / confidence) and InferIntentNode exist in a branch but aren't in the main pipeline yet.
Recipe as First-class Entity
Planned
Currently emergent from RoutingRule + StrategyConfig rows. No version history, no diff view, no clone-for-new-client. A dedicated Recipe model would make this manageable at scale.
RoutingRule Not Account-Scoped
In-progress fix
Current RoutingRule has no CapitalAccount FK — rules are team-level. The planned IntentRoutingRule model carries a bookkeeping_account FK. Until merged, a matching rule in one account's feed would match another account's feed.
expense_split Strategy
Reserved slot
Constant defined in RoutingModule.CHOICES. No concrete config model, no handler, no workflow node. Itemised receipt matching is the intended use case.
Sub-workflow Invocation
Planned
Nodes are currently copied explicitly across workflows. A SubWorkflowNode pattern would let a parent invoke a named child execution and merge context on completion — avoiding node duplication.
GL Context Size
Planned
Large GL files (>10K rows) stored inline in workflow context JSON. Should reference GCS object storage URL + metadata only. Known limitation for CSV onboarding path.
